## 📋 文档更新原则

### 核心原则
1. **代码未动，文档先行** - 修改代码前先更新相关文档
2. **谁开发，谁文档** - 功能开发者负责对应文档
3. **每日审查** - TL 每日检查文档更新情况
4. **网站同步** - 开发完成后立即更新网站文档

## 🔄 文档更新流程

### 开发前
```
1. 查看相关文档是否完整
2. 如有缺失，先创建文档框架
3. TL 确认文档结构
```

### 开发中
```
1. 记录关键技术决策
2. 更新 API 接口变更
3. 记录遇到的问题和解决方案
```

### 开发后
```
1. 更新功能文档（10 分钟内完成）
2. 提交文档到 Git（如适用）
3. 上传到网站文档中心
4. TL 审查文档质量
```

## 📅 文档审查机制

### 每日审查（TL 负责）
- **时间**: 每日 17:00
- **内容**: 
  - 检查项目进度看板是否更新
  - 检查新增功能是否有文档
  - 检查文档是否与代码一致
- **输出**: 文档审查报告

### 每周审查（全员）
- **时间**: 每周日 18:00
- **内容**:
  - 审查各自负责的文档
  - 更新过时内容
  - 删除废弃文档
- **输出**: 周文档维护报告

# 文档同步脚本
# 用法：./sync-docs.sh
